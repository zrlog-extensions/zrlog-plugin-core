# ZrLog Plugin Runtime v4.0 收敛与后续实现计划

本文是 `docs/v4.0 zrlog-plugin-runtime.md` 的实施收敛版。

原文档保留为完整设计背景，后续开发以本文作为任务边界和验收清单。

## 1. v4.0 收敛目标

v4.0 只做轻量 Plugin Runtime 的第一阶段闭环：

1. 插件能力统一描述为 capability。
2. plugin-core 负责 capability 注册、调度、通知路由、运行态和调用日志。
3. 业务插件继续拥有自己的业务配置和业务行为。
4. runtime 数据继续存 website KV，不新增数据库表。
5. 插件进程支持按需启动和空闲回收。
6. scheduler 可以触发 reminder 的 scheduled capability。
7. reminder 通过标准 notification request 调用 email notification channel。
8. Native Image 仍走 GraalVM Agent warmup，不维护手写 reflect/resource 配置。

## 2. v4.0 不再扩大的范围

以下内容不进入 v4.0 第一阶段：

1. 完整 MCP 插件或 MCP 协议实现。
2. 通用事件总线。
3. Quartz 完整 Scheduler。
4. 新增业务表。
5. plugin-core 托管 SMTP、邮件模板或 reminder 业务配置。
6. HTTP notification publish 入口替代插件 socket publish。
7. 只按 capability key 或 service name 的全局快捷调用入口。
8. 独立 `runtime.audit` / `runtime.security` 包的完整策略体系。

`runtime.audit` 和 `runtime.security` 在 v4.0 只保留为后续方向。当前阶段用 invocation log、exposure 校验和 token 校验解决实际风险。

## 3. 当前已落地内容

### 3.1 plugin-common

已具备：

1. `@Capability`
2. `@ScheduledCapability`
3. `PluginCapability`
4. `CapabilityInvokeRequest`
5. `CapabilityInvokeResult`
6. `NotificationRequest`
7. `SchedulerQueryRequest`
8. `SchedulerQueryResult`
9. `SchedulerUpdateRequest`（历史兼容，只用于废弃入口失败响应）
10. `SchedulerUpdateResult`（历史兼容，只用于废弃入口失败响应）
11. `ActionType.CAPABILITY_INVOKE`
12. `ActionType.NOTIFICATION_PUBLISH`
13. `ActionType.SCHEDULER_QUERY`
14. `ActionType.SCHEDULER_UPDATE`（历史兼容，只读拒绝写入）
15. `BasicCronParser`
16. `CronParseException`
17. `NioClient` 读取 service 注解并随 `INIT_CONNECT` 上报 capabilities

### 3.2 plugin-core

已具备：

1. `KvRepository` / `WebsiteRuntimeKvStore`
2. capability 存储、注册和查询
3. scheduler automation、run log、internal runner、external tick endpoint
4. Bearer token 校验
5. runtime states、按需启动、空闲回收
6. notification channel provider 查找和 delivery log
7. runtime event handler 查找和按需拉起，`refreshCache()` 已通过 `system.refreshCache` 发布事件
8. service provider 选择页面和 API
9. runtime 页面路由：
   - `runtime-scheduler`
   - `runtime-states`
   - `runtime-notification`
   - `runtime-services`
10. runtime API 单段命名，避免 `/runtime/*` 被插件代理误判为插件名
11. `status()` 在按需加载模式下不再等待全部插件启动

### 3.3 runtime event

runtime event 是 plugin-core 面向系统事件的最小事件分发层，不替代 notification。

当前已支持：

1. `type=event_handler`
2. `exposure` 包含 `runtime_event`
3. `channel` 暂时承载订阅事件名，例如 `system.refreshCache`
4. `channel` 支持逗号分隔多个事件名
5. `CapabilityInvoker` 调用时使用 `InvokeContext.source=runtime_event`
6. 调用前仍走统一 capability 校验、按需拉起、invocation log
7. `PluginApiController.refreshCache()` 先发布 `system.refreshCache`，再对没有声明 handler 的已运行插件发送旧 `ActionType.REFRESH_CACHE`

插件声明示例：

```java
@Capability(
    key = "plugin.cache.refresh",
    type = "event_handler",
    exposure = {"runtime_event"},
    channel = "system.refreshCache"
)
```

兼容说明：

1. `system.refreshCache` 的别名包含 `refresh_cache` 和 `REFRESH_CACHE`。
2. 当前 `PluginCapability` 线模型还没有 `eventTypes` 字段，所以第一阶段复用 `channel`。
3. 后续 plugin-common 增加 `eventTypes` 后，resolver 可以同时读取 `eventTypes` 和 `channel`，不影响已声明插件。
4. 旧 `ActionType.REFRESH_CACHE` 保留，只作为已运行旧插件的兼容广播，不再负责拉起插件。

### 3.4 email / reminder 插件

已具备：

1. email 插件声明 `notification.email.send`
2. email 插件保留旧 `emailService`
3. reminder 插件声明 `reminder.scanDueTasks`
4. reminder 到期扫描通过 `NotificationRequest` 发布通知
5. reminder 等待通知 publish 响应，不把 dispatch 当完成

## 4. 剩余计划和最新状态

### 4.1 CapabilityInvoker 边界已完成

目标是所有跨插件 capability 调用都经过同一套校验和日志链路。

状态：已完成。统一校验已进入 `TrackingCapabilityInvoker`，`SchedulerRuntime`、`NotificationRuntime` 和 internal capability 调用都走 `RuntimeCapabilityInvokerFactory.socket(kvStore)` 返回的完整调用链。

已处理：

1. `TrackingCapabilityInvoker` 统一负责 capability 存在、enabled、exposure、目标插件可启动校验。
2. 调度、通知和 internal capability 调用不再各自维护完整校验。
3. `ServerActionHandler.capabilityInvoke()` 会拒绝 disabled capability。
4. 调用失败仍写 invocation log。

### 4.2 reminder notification channels 已完成

目标是 reminder 只配置逻辑通知渠道，不配置 email 实现。

状态：已完成。reminder 已新增 `plugin.reminder.notification.channels`，没有 setting 时默认 `["email"]`，高优先级任务使用 `importantChannels`，单条任务的旧 `emailNotify` 仅作为“是否通知”的兼容开关。

已处理：

1. reminder 发送通知按 channels setting 决定 channel。
2. reminder 新增 `plugin.reminder.notification.channels` 存储。
3. 前端和插件卡片表单文案从“邮件提醒”收敛为“通知提醒”。

### 4.3 notification 配置页面还缺测试通知

目标是 notification tab 可验证 provider 是否可用。

当前仍存在的问题：

1. 没有发送测试通知入口。
2. provider 列表没有聚合展示最近 delivery 状态。
3. `lastDeliveryStatus`、`lastDeliveryAt`、`updatedAt` 等字段还没补齐。

### 4.4 GraalVM Agent warmup 还不完整

目标是新增 runtime model、ActionType 和注解读取路径都被 agent 覆盖。

当前仍存在的问题：

1. `GraalvmAgentApplication` 主要注册 model class，没有驱动完整 action 分发路径。
2. `CAPABILITY_INVOKE`、`NOTIFICATION_PUBLISH`、`SCHEDULER_QUERY`、`SCHEDULER_UPDATE` 的基础路径没有 warmup；其中 `SCHEDULER_UPDATE` 只应覆盖废弃入口返回失败。
3. `NotificationProviderSetting` 需要显式进入 Gson/native 注册。
4. plugin-common 的注解读取和 wire model 需要继续确认 agent 覆盖。

## 5. 下一步实现顺序

### P0-1 收口 CapabilityInvoker

状态：已完成。统一校验已进入 `TrackingCapabilityInvoker`，`RuntimeCapabilityInvokerFactory.socket(kvStore)` 返回完整调用链，调度、通知和 internal capability 调用不再各自维护完整校验。

实现目标：

1. 在 plugin-core 内新增统一的 capability 调用边界。
2. 调用前必须校验：
   - `pluginId` 必填
   - `capabilityKey` 必填
   - capability 存在
   - capability enabled
   - exposure 包含 `InvokeContext.source`
   - 目标插件存在且可启动
3. 调用链必须保持：
   - `ensureStarted(pluginId)`
   - `markInvocationStart(pluginId)`
   - socket `CAPABILITY_INVOKE`
   - `markInvocationEnd(pluginId)`
   - append `plugin.runtime.invocationLogs`
4. `SchedulerRuntime`、`NotificationRuntime`、`ServerActionHandler.capabilityInvoke()` 都走同一个边界。
5. 不新增只按 key 调用的核心接口。

建议实现方式：

1. 保留 `SocketCapabilityInvoker` 只做 socket 通信。
2. 将 `TrackingCapabilityInvoker` 扩展为完整 runtime invoker，或新增 `CheckedCapabilityInvoker` 包装在 tracking 前。
3. `RuntimeCapabilityInvokerFactory.socket(kvStore)` 返回完整链路，不暴露半成品 invoker。
4. `InvokeContext.source` 使用固定值：
   - scheduler
   - notification
   - internal
   - admin_ui

必须补测试：

1. capability 不存在时不调用 socket，写 invocation log。
2. disabled capability 不调用 socket，写 invocation log。
3. exposure 不匹配时不调用 socket，写 invocation log。
4. plugin 无法启动时返回失败并写 invocation log。
5. scheduler 和 notification 不再重复维护完整校验逻辑。
6. internal capability 调用会拒绝 disabled capability。

### P0-2 完成 reminder channel 设置

状态：已完成。实现位于 `zrlog-plugin-reminder`，任务级字段继续兼容 `emailNotify`，但语义收敛为是否发送通知。

实现目标：

1. reminder 新增 notification channel setting。
2. 存储 key：

```text
plugin.reminder.notification.channels
```

3. 默认值：

```json
{
  "schema": "plugin.reminder.notification.channels",
  "version": 1,
  "data": {
    "defaultChannels": ["email"],
    "importantChannels": ["email"],
    "failedChannels": ["email"]
  }
}
```

4. reminder publish notification 时根据 setting 决定 channels。
5. 单条任务仍可保留“是否通知”的开关，但文案和字段语义应从 email 收敛为 notification。
6. 不在 reminder 中配置 SMTP、email provider 或 email 插件内部实现。

建议迁移：

1. 旧 `emailNotify=true` 表示该任务允许通知。
2. 新渠道默认从 `defaultChannels` 读取。
3. 没有 setting 时使用 `["email"]`，保证升级后行为不变。
4. UI 文案从“邮件提醒”改为“到期通知”或“通知提醒”。

必须补测试：

1. 没有 setting 时默认发 email channel。
2. setting 指定多个 channels 时按配置发布。
3. 任务关闭通知时不 publish。
4. notification publish 失败时不标记 reminded。

### P1-1 补 notification 测试通知和最近状态


## 6. 验收标准

必须通过：

```bash
mvn -q test
```

如果只改了 runtime 小范围逻辑，可以先跑：

```bash
mvn -q -Dtest=TrackingCapabilityInvokerTest,SchedulerRuntimeTest,NotificationRuntimeTest test
```

再跑完整测试。

必须确认：

1. `PluginCapability.serviceName` 可用。
2. `NioClient` 会读取 `@Capability` 和 `@ScheduledCapability`。
3. `PluginNativeImageUtils.usedGsonObject()` 覆盖新增 wire model。
4. common 可被 plugin-core、email、reminder 使用。

建议命令：

```bash
mvn -q -DskipTests install
```

必须确认：

## 7. 端到端验收场景

完成 P0 后，至少手工验证一次：

1. 启动 plugin-core，按需加载开启。
2. email 插件未启动时，notification test 可以按需拉起 email。
3. reminder 插件未启动时，scheduler due automation 可以按需拉起 reminder。
4. reminder 扫描到到期任务。
5. reminder 发布 `NOTIFICATION_PUBLISH`。
6. plugin-core 找到 `notification.email.send`。
7. plugin-core 通过统一 CapabilityInvoker 调用 email。
8. email 返回 `CapabilityInvokeResult.success=true`。
9. `plugin.runtime.notificationDeliveries` 有成功记录。
10. `plugin.runtime.invocationLogs` 有 scheduler/reminder/email 相关调用记录。
11. 空闲超时后本机插件进程退出，运行态实例列表移除对应实例。

## 8. 不能破坏的约束

1. 不新增数据库表。
2. runtime 数据只能写 website KV 或既有 `plugin_core_db_key`。
3. plugin-core 不能读取或写入 email 插件 SMTP 配置。
4. reminder 不能直接调用 email 插件内部类。
5. scheduler、notification、未来 MCP 都不能直接拿 IOSession 调业务插件。
6. 生产环境未登录访问插件非公开路径必须返回 403。
7. `DEV_MODE` 请求头不能作为生产权限绕过条件。
8. Native Image 兼容不能只靠普通 JVM 测试推断。
