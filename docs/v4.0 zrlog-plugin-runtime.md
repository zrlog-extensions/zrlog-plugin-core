# ZrLog Plugin Runtime 简洁化升级执行方案

## 0. 背景

ZrLog 当前插件系统已经是进程外插件模型，而不是 JVM 内嵌插件模型。

现有事实：

1. `zrlog-plugin-core` 启动 `NioServer`，插件通过 TCP 连接回来。
2. 插件由 `PluginScanRunnable` 以独立进程方式启动，支持 `.jar`、`.bin`、`.exe`。
3. `zrlog-plugin-common` 的 `NioClient` 读取 `plugin.properties` 后发起 `INIT_CONNECT` 注册插件。
4. 当前已有 `paths`、`actions`、`services`、`dependentService` 等概念。
5. 插件数据当前主要存储在 `website` KV 表中。
6. 当前阶段不得新增业务表，必须保持 website KV 存储设计。
7. ZrLog 定位是简洁好用，不做大而全插件平台。

本次改造目标不是重写插件系统，而是把现有能力收口为更清晰的插件运行时：

```text
Plugin
Capability
Scheduler
Notification
Runtime
```

最终目标：

```text
插件是进程外运行的能力集合；
plugin-core 是插件运行时；
插件不自带长期 timer；
定时任务由 plugin-core 统一调度；
通知通道标准化，但具体实现仍由通知插件负责；
未来 MCP 作为独立插件读取 capability，不进入 plugin-core 主逻辑。
```

---

## 1. 总体目标

本阶段目标：

1. 保持现有 TCP/MsgPacket/IOSession 通讯模型。
2. 保持现有 website KV 存储模型。
3. 不新增数据库表。
4. 不重写插件启动机制。
5. 不引入复杂 MCP 实现。
6. 不把业务逻辑塞进 plugin-core。
7. 先完成 capability、scheduler、notification 的最小闭环。
8. 后续能平滑升级到 MCP、事件触发、更多通知通道。

---

## 2. 非目标

本阶段不做：

1. 不做新数据库表。
2. 不做分布式调度。
3. 不做复杂插件进程池。
4. 不做多语言 SDK。
5. 不重写 TCP 协议。
6. 不做完整 MCP 插件。
7. 不做大而全 AI 插件。
8. 不把发布文章流程下放给插件。
9. 不把资源库功能插件化。
10. 不把 email 发送逻辑回收进 plugin-core。

---

## 3. 核心设计原则

### 3.1 plugin-core 只管秩序

plugin-core 负责：

```text
插件注册
插件状态
插件 capability 注册
插件 capability 调用
插件定时任务调度
插件执行日志
通知通道发现
通知标准入口
```

plugin-core 不负责：

```text
reminder 业务
email SMTP 发送
backup 备份细节
static-plus 同步细节
site-check 爬取细节
MCP 协议细节
```

### 3.2 插件只管业务能力

业务插件负责：

```text
声明 capability
实现 capability
声明通知事件
配置自己支持的通知渠道
```

业务插件不得：

```text
自行启动长期 timer
直接依赖 email 插件内部实现
直接调用其他插件进程
绕过 plugin-core 写统一执行日志
```

### 3.3 继续使用 website KV

当前阶段不得新增表。

所有新增数据必须存储到 `website` KV 表。

必须通过 Store 封装访问，禁止 runtime 逻辑直接散写 `WebSiteDAO`。

## 4. 存储规范

### 4.1 KV key 规范

plugin-core runtime 数据统一使用：

```text
plugin.runtime.*
```

具体插件业务配置使用：

```text
plugin.{shortName}.*
```

例如：

```text
plugin.runtime.settings
plugin.runtime.capabilities
plugin.runtime.automations.v2
plugin.runtime.automationRuns.v2
plugin.runtime.states
plugin.runtime.notification.channels
plugin.runtime.notificationDeliveries
plugin.runtime.invocationLogs
plugin.runtime.mcp.policy

plugin.reminder.settings
plugin.reminder.notification.channels
plugin.email.settings
plugin.staticPlus.settings
```

说明：

```text
plugin.runtime.settings 仅作为未来通用 runtime 设置预留。
scheduler 调度中心配置第一阶段沿用 PluginCore.setting，由现有 plugin_core_db_key 持久化。
```

### 4.2 JSON value 规范

所有 runtime KV value 必须是 JSON object。

统一结构：

```json
{
  "schema": "plugin.runtime.automations.v2",
  "version": 2,
  "updatedAt": "2026-05-29T00:00:00Z",
  "items": []
}
```

必须包含：

```text
schema
version
updatedAt
items
```

如果是单对象配置，也使用：

```json
{
  "schema": "plugin.runtime.settings",
  "version": 1,
  "updatedAt": "2026-05-29T00:00:00Z",
  "data": {}
}
```

### 4.3 日志保留策略

前期数据量不大，日志可以集中存一个 KV key。

使用：

```text
plugin.runtime.automationRuns.v2
plugin.runtime.notificationDeliveries
plugin.runtime.invocationLogs
```

每个 key 中最多保留：

```text
automationRuns: 500 条
notificationDeliveries: 500 条
invocationLogs: 1000 条
```

append 新日志时必须裁剪旧数据。

未来数据量增大时，可以平滑迁移为：

```text
plugin.runtime.automationRuns.20260529
plugin.runtime.notificationDeliveries.20260529
plugin.runtime.invocationLogs.20260529
```

或者迁移到独立表。

### 4.4 Store 抽象

必须新增 Store 接口。

禁止在 SchedulerRuntime、NotificationRuntime、CapabilityInvoker 中直接操作 `WebSiteDAO`。

通用 KV 抽象放在 `zrlog-plugin-common`：

```text
com.zrlog.plugin.common
```

接口：

```java
public interface KvRepository {
    Optional<String> get(String key);
    void put(String key, String value);
}
```

plugin-core 保留基于 website 表的实现：

```java
public class WebsiteRuntimeKvStore implements KvRepository {
    // 内部使用 WebSiteDAO
}
```

再基于 KvRepository 实现：

```java
CapabilityStore
AutomationStore
NotificationStore
InvocationLogStore
RuntimeSettingStore
```

## 5. Capability 设计

### 5.1 目标

统一当前 `actions`、`services`、`paths` 等概念，形成插件能力模型。

第一阶段不废弃旧字段，只新增 capability 层。

### 5.2 模型

新增：

```java
public class PluginCapability {
    private String pluginId;
    private String pluginName;
    private String key;
    private String type;
    private String label;
    private String description;
    private List<String> exposure;
    private String riskLevel;
    private Boolean readOnly;
    private Boolean requiresConfirmation;
    private Integer timeoutSeconds;
    private Integer concurrency;
    private Boolean enabled;
    private Boolean legacy;
    private Boolean generated;
}
```

说明：

```text
pluginId 是插件身份 ID，沿用插件协议中的 Plugin.id。
pluginId 用于 runtime state、capability、scheduler、service provider、notification provider、local session 和进程管理等运行时索引。
pluginId 不表示本次启动实例，也不表示某条 socket session。
shortName 是插件短名，在 ZrLog 内唯一，用于插件包文件名、路由段、插件列表 map key 和展示/查找入口。
pluginName 是插件协议中的 Plugin.name，只表示人类可读名称；为空时宿主可以用 shortName 作为 UI 文案兜底，不能再新增等价名称字段。
插件管理入口、插件文件扫描、插件页面路由、停止、卸载等自然接收 shortName 的场景，应先按 shortName 定位当前插件，再在 runtime state、capability、scheduler、service/notification provider 等持久化运行时数据中使用 pluginId。
runtimeInstanceId / sessionId 表示一次进程或 socket 会话的实例身份，用于同一 pluginId 下多进程、多 session 状态聚合。
pluginId + key 是 runtime 内部唯一标识时，pluginId 必须使用 Plugin.id。
key 推荐全局语义唯一，但第一阶段不强制全局唯一。
所有真实调用必须带 pluginId，避免沿用旧 service 全局按 name 查找带来的歧义。
legacy=true 表示由旧 @Service 兼容生成。
generated=true 表示不是插件显式声明的 capability。
```

### 5.3 type

支持：

```text
service
scheduled
notification_channel
event_handler
resource
action
```

第一阶段重点支持：

```text
service
scheduled
notification_channel
event_handler
```

### 5.4 exposure

支持：

```text
internal
admin_ui
scheduler
notification
runtime_event
mcp
```

第一阶段重点支持：

```text
internal
scheduler
notification
runtime_event
```

### 5.5 riskLevel

支持：

```text
low
medium
high
critical
```

默认：

```text
low
```

### 5.6 key 命名

必须使用点分命名：

```text
reminder.scanDueTasks
notification.email.send
notification.webhook.send
siteCheck.run
backup.database
staticPlus.sync
statistics.dailyReport
```

禁止使用：

```text
run
send
handle
process
doTask
```

### 5.7 key 唯一性

第一阶段不强制 `capability.key` 全局唯一。

规则：

```text
1. registry 存储和调用以 pluginId + key 为准。
2. admin 查询可以按 key 聚合，但执行必须明确 pluginId。
3. 如果多个插件声明同一个 key，registry 标记为 ambiguous，不允许通过只传 key 的快捷入口调用。
4. 面向 MCP 或其他外部自动化入口时，ambiguous key 默认不可暴露。
5. 推荐官方插件使用稳定的点分命名，避免未来迁移成本。
```

示例：

```text
reminder.scanDueTasks
mail.notification.send
webhook.notification.send
```

允许不同插件短期存在同名兼容 service，但新 capability 不应主动制造同名 key。

## 6. plugin-common 扩展

### 6.1 保留 @Service

当前已有 `@Service` 和 `IPluginService`，第一阶段继续复用。

### 6.2 新增 @Capability

在 `zrlog-plugin-common` 中新增注解：

```java
@Retention(RetentionPolicy.RUNTIME)
public @interface Capability {
    String key();

    String type() default "service";

    String label() default "";

    String description() default "";

    String[] exposure() default {"internal"};

    String riskLevel() default "low";

    boolean readOnly() default false;

    boolean requiresConfirmation() default false;

    int timeoutSeconds() default 30;

    int concurrency() default 1;
}
```

### 6.3 新增 @ScheduledCapability

```java
@Retention(RetentionPolicy.RUNTIME)
public @interface ScheduledCapability {
    String key();

    String label() default "";

    String description() default "";

    String defaultCron() default "";

    String timezone() default "system";

    int timeoutSeconds() default 60;

    int concurrency() default 1;
}
```

`@ScheduledCapability` 是 `@Capability` 的便捷声明。

默认等价于：

```text
type = scheduled
exposure = scheduler
timeoutSeconds = 60
concurrency = 1
```

如果 scheduled capability 声明了 `defaultCron`，plugin-core 在插件 `INIT_CONNECT` 注册 capability 后，会自动为该 pluginId + capabilityKey 创建一条默认 automation。
如果调度中心已经存在同一 pluginId + capabilityKey 的 automation，则保留用户已有 cron、启用状态和 payload，不覆盖。

如果同一个 service class 同时声明 `@Capability` 和 `@ScheduledCapability`：

```text
1. key 必须一致。
2. @Capability 提供通用元信息。
3. @ScheduledCapability 提供 defaultCron、timezone 等调度默认值。
4. type 和 exposure 以 scheduled 语义为准，不允许被覆盖成非 scheduler 能力。
```

### 6.4 兼容规则

1. 旧插件只使用 `@Service`，仍然可运行。
2. 新插件推荐同时使用 `@Service` 和 `@Capability`。
3. 没有 `@Capability` 的旧 service 仍然只能通过旧 `SERVICE` 路径调用。
4. 只有 exposure 包含 `scheduler` 的 capability 才能被定时任务调用。
5. 只有 type 为 `notification_channel` 的 capability 才能作为通知通道。

### 6.5 capability 注册报文

第一阶段不让 plugin-core 扫描插件 class。

`zrlog-plugin-common` 负责在插件进程内读取注解，并把 capability 元信息随 `INIT_CONNECT` 一起发送给 plugin-core。

实现方式：

```text
1. 在 zrlog-plugin-common 中新增 wire model：com.zrlog.plugin.message.PluginCapability。
2. 在 Plugin 中新增 List<PluginCapability> capabilities。
3. NioClient 遍历 serviceList 时读取 @Service、@Capability、@ScheduledCapability。
4. NioClient 将显式声明的 capability 写入 Plugin.capabilities。
5. plugin-core 在 ServerActionHandler.initConnect() 中注册这些 capabilities。
```

这样保持进程外模型，不要求 plugin-core 拿到插件 classpath，也不新增 capability 注册 ActionType。

### 6.6 旧 service 兼容过滤

旧插件没有 capability 声明时，不自动把所有 service 都暴露成新 capability。

兼容策略：

```text
1. 旧 @Service 保持原行为，仍然通过 ActionType.SERVICE 可调用。
2. capability registry 可以生成兼容视图，但只生成 internal + legacy + generated 能力。
3. 兼容视图默认过滤掉不符合点分 key 命名的 service。
4. 兼容生成的 capability 不允许 scheduler、notification、mcp exposure。
5. 兼容生成的 capability 不作为插件显式能力持久化。
```

过滤规则：

```text
service name 必须匹配：^[a-z][a-zA-Z0-9]*(\.[a-z][a-zA-Z0-9]*)+$
```

不符合规则的旧 service：

```text
1. 继续通过旧 SERVICE 路径可用。
2. 不出现在默认 capability 列表。
3. 可在诊断信息中标记为 legacyServiceSkipped。
```

移除条件：

```text
当官方插件完成 capability 显式声明，并且最低支持的 zrlog-plugin-common 版本进入 v4 后，可以移除兼容生成逻辑。
```

## 8. CapabilityInvoker

### 8.1 职责

所有跨插件能力调用必须经过 CapabilityInvoker。

禁止 scheduler、notification、mcp 插件直接拿 IOSession 调业务插件。

接口：

```java
public interface CapabilityInvoker {
    CapabilityInvokeResult invoke(String pluginId, String capabilityKey, Map<String, Object> payload, InvokeContext context);
}
```

第一阶段 `pluginId` 必填。

不提供只按 key 调用的核心接口，避免旧 service 全局查找语义继续扩散。

### 8.2 InvokeContext

```java
public class InvokeContext {
    private String source;
    private String userId;
    private String requestId;
    private String traceId;
    private boolean auditRequired;
}
```

source 支持：

```text
scheduler
notification
admin_ui
internal
mcp
```

### 8.3 调用流程

必须按以下顺序：

```text
1. 查询 capability
2. 检查 capability.enabled
3. 检查 exposure 是否包含 context.source
4. 检查插件是否 enabled
5. ensureStarted(pluginId)
6. activeInvocationCount + 1
7. 通过新的 CAPABILITY_INVOKE/MsgPacket 机制调用插件
8. 等待响应或超时
9. activeInvocationCount - 1
10. 记录 invocation log
11. 返回结果
```

第一阶段插件端可以复用现有 `@Service` 实现。

映射：

```text
capability.key = @Service.value
```

但 socket 层不继续复用 `ActionType.SERVICE` 作为新 capability 调用入口。

### 8.4 新 ActionType

在 `zrlog-plugin-common` 中新增：

```text
CAPABILITY_INVOKE
NOTIFICATION_PUBLISH
SCHEDULER_QUERY
SCHEDULER_UPDATE
```

方向：

```text
CAPABILITY_INVOKE: plugin-core -> plugin
NOTIFICATION_PUBLISH: plugin -> plugin-core
SCHEDULER_QUERY: plugin -> plugin-core
SCHEDULER_UPDATE: plugin -> plugin-core
```

`SCHEDULER_QUERY` 是只读查询入口，插件只能查询当前 IOSession 绑定插件自身的 scheduled automation。
可返回 cron、enabled、timezone、nextRunAt、lastRunAt 等用于插件页面展示的字段。

`SCHEDULER_UPDATE` 只作为历史兼容 ActionType 保留，不再承载插件侧写入调度配置。
插件不允许通过 socket 内部消息修改 cron、启停、nextRunAt 或 payload。
这些字段以调度中心为唯一写入口。

不新增：

```text
CAPABILITY_REGISTER
```

原因：

```text
capability 元信息已经随 INIT_CONNECT 的 Plugin.capabilities 兼容注册。
```

### 8.5 CAPABILITY_INVOKE

core 调用插件 capability 时发送：

```java
// com.zrlog.plugin.message.CapabilityInvokeRequest
public class CapabilityInvokeRequest {
    private String pluginId;
    private String capabilityKey;
    private String source;
    private String requestId;
    private String traceId;
    private Map<String, Object> payload;
}
```

插件端处理：

```text
1. ClientActionHandler 接收 CAPABILITY_INVOKE。
2. 根据 capabilityKey 找到插件进程内对应的 @Service class。
3. 复用 IPluginService.handle(...) 执行业务。
4. 返回 CapabilityInvokeResult。
```

插件也可以通过 `IOSession.requestCapability(...)` 向 plugin-core 发起 `CAPABILITY_INVOKE`。
server 侧只转发已经注册、且 exposure 包含 `internal` 的 capability，并统一写 runtime invocation log。

旧 `ActionType.SERVICE` 继续保留给旧插件和旧跨插件调用路径。

### 8.6 SCHEDULER_QUERY

插件需要在只读页面展示自身 scheduled capability 的调度状态时，通过 `SCHEDULER_QUERY` 查询 plugin-core。

请求：

```java
// com.zrlog.plugin.message.SchedulerQueryRequest
public class SchedulerQueryRequest {
    private String capabilityKey;
}
```

响应：

```java
// com.zrlog.plugin.message.SchedulerQueryResult
public class SchedulerQueryResult {
    private boolean success;
    private String errorMessage;
    private String automationId;
    private String capabilityKey;
    private String name;
    private String cron;
    private String timezone;
    private Boolean enabled;
    private String nextRunAt;
    private String lastRunAt;
}
```

规则：

```text
1. plugin-core 使用当前 IOSession 绑定的 pluginId，不接受插件在 request 中指定 pluginId。
2. 只能查询本插件已注册的 type=scheduled 且 exposure 包含 scheduler 的 capability。
3. 查询只读，不会创建、修改或补全 automation。
4. automation 不存在时返回失败，由调度中心或 INIT_CONNECT 默认任务创建流程处理。
5. nextRunAt、lastRunAt 返回 epoch millis 字符串，插件页面按本地时区格式化展示。
```

插件端可使用：

```java
session.querySchedule("statistics.dailyReport", callback);
```

### 8.7 SCHEDULER_UPDATE

`SCHEDULER_UPDATE` 是废弃的历史兼容入口。
服务端收到该 ActionType 后必须返回失败，不得创建或修改 automation。

收口规则：

```text
1. 子插件只通过 @ScheduledCapability 声明可调度业务能力和 defaultCron。
2. plugin-core 在 INIT_CONNECT 后按 pluginId + capabilityKey 创建或保留默认 automation。
3. cron、启停、payload、立即执行和删除只能在调度中心页面/API 中修改。
4. 插件业务页面只展示业务数据和任务状态，需要调度状态时走 SCHEDULER_QUERY。
5. 统计/PV 这类报表插件是只读报表；需要编辑调度时跳转到调度中心定位对应任务。
6. 旧插件调用 IOSession.updateSchedule* 时只会收到失败响应，不再影响调度中心状态。
```

保留 `SchedulerUpdateRequest` / `SchedulerUpdateResult` wire model 仅用于旧插件失败响应和 native-image 兼容。

### 9.4 tick endpoint

tick 是 scheduler 的一次调度心跳。
每次 tick 只检查并执行 `enabled=true` 且 `nextRunAt <= now` 的 automation。
tick 不会执行未到期任务，也不会忽略 automation 的 enabled 状态。
如果需要绕过排期马上跑某一条任务，应使用调度中心页面行内的“立即执行”，或调用 `POST /api/runtime-automations/run`。

正常情况下不需要人工或外部系统频繁调用 tick。
plugin-core 启动后会启动轻量内部 tick runner，默认每 60 秒自动检查一次 due automation。

需要使用 tick 的场景：

```text
1. 调度中心页面手动验证当前是否有 due automation。
2. 部署、重启或插件重连后，想立即补一次 due 检查，不等内部 runner 的下一轮。
3. 接入独立调度中心或外部 cron，希望由外部系统驱动 plugin-core 执行到期任务。
4. 排查 scheduler 时，需要确认任务发现、执行、automationRuns 写入链路是否正常。
```

不应该使用 tick 的场景：

```text
1. 想立即执行一个还没到 nextRunAt 的任务。
2. 想忽略 disabled 状态强制执行任务。
3. 想绕过 capability exposure 或插件状态校验。
```

调度中心页面使用的内部管理 API：

```http
POST /api/runtime-scheduler/tick
POST /p/api/runtime-scheduler/tick
POST /plugin/api/runtime-scheduler/tick
POST /admin/plugins/api/runtime-scheduler/tick
```

该 API 不需要外部 token，面向 plugin-core 页面自身使用。

外部调度器使用的 HTTP endpoint：

```http
POST /api/internal/plugin/scheduler/tick
```

调度中心页面展示的 provider secret 只用于外部 tick：

```text
Authorization: Bearer <secret>
```

如果只使用 plugin-core 内部 runner，或只在页面点击“检查到期任务”，不需要复制和使用 secret。

通过 ZrLog 插件代理访问时：

```http
POST /p/api/internal/plugin/scheduler/tick
POST /plugin/api/internal/plugin/scheduler/tick
```

说明：

```text
zrlog-blog-web 的 BlogPluginInterceptor 会拦截 /plugin/* 和 /p/*，并转发给 PluginCorePlugin.accessPlugin。
plugin-core 实际收到的 URI 是去掉 /plugin/ 或 /p/ 后的内部 URI。
```

职责：

```text
1. 校验 Bearer token
2. 读取 `plugin.runtime.automations.v2`
3. 找到 enabled=true 且 nextRunAt <= now 的任务
4. 检查 capability 是否允许 scheduler 调用
5. 调用 CapabilityInvoker
6. 写 `plugin.runtime.automationRuns.v2`
7. 更新 lastRunAt 和 nextRunAt
```

页面内“立即执行”单条任务 API：

```http
POST /api/runtime-automations/run
```

参数：

```text
id=<automationId>
```

### 9.5 外部 tick 认证

请求头：

```http

```

认证规则：

```text
1. 从 Authorization 请求头读取 Bearer token。
2. token 必须等于某个 enabled provider 的 secret。
3. 不再要求 timestamp、rawBody 和签名计算。
```

外部调度器调用示例：

```bash

  -H "Authorization: Bearer ${secret}" \
  https://blog.example.com/api/internal/plugin/scheduler/tick
```

通过插件代理调用示例：

```bash

  -H "Authorization: Bearer ${secret}" \

```

调度中心页面提供 `外部 Host` 配置，用来生成可直接复制的完整 curl：

```text

```

规则：

```text
1. externalHost 为空时，后端读取 loadPublicInfo 的 homeUrl 作为默认 host。
2. externalHost 可以填写 https://blog.example.com，也可以填写 blog.example.com；未带协议时按 https:// 补齐。
3. 最终调用路径固定为 /api/internal/plugin/scheduler/tick。
4. Cloudflare Worker、系统 crontab、GitHub Actions 等外部调度器都应使用完整 externalTickUrl。
```

Cloudflare Timer Worker 要定时执行的是同一个 `POST /api/internal/plugin/scheduler/tick` 请求。Worker 示例：

```js
export default {
  async scheduled(controller, env, ctx) {
    ctx.waitUntil(tick(env));
  },

  async fetch() {
    return new Response("ZrLog scheduler worker is running. Cron uses scheduled().", {
      headers: {"content-type": "text/plain; charset=utf-8"},
    });
  },
};

async function tick(env) {
  const response = await fetch(env.ZRLOG_SCHEDULER_TICK_URL, {
    method: "POST",
    headers: {
      Authorization: `Bearer ${env.ZRLOG_SCHEDULER_SECRET}`,
    },
  });

  if (!response.ok) {
    throw new Error("ZrLog scheduler tick failed: " + response.status + " " + await response.text());
  }
}
```

`scheduled()` 是定时执行入口；`fetch()` 只用于 Worker 地址健康检查，避免打开 Worker 时报 `No fetch handler`。

`wrangler.toml` 示例：

```toml

compatibility_date = "2026-05-30"

[triggers]
crons = ["*/5 * * * *"]

[vars]

```

secret 使用 Cloudflare Worker Secret 保存：

```bash

```

### 9.6 设置存储

scheduler 设置沿用当前 plugin-core 配置对象。

当前 `PluginCoreDAO` 已经将 `PluginCore` 持久化到 website KV 的 `plugin_core_db_key` 中，因此 scheduler 调度中心配置第一阶段放入：

```text
PluginCore.setting.scheduler
```

示例：

```json
{
  "disableAutoDownloadLostFile": false,
  "scheduler": {
    "enabled": true,
    "externalHost": "https://blog.example.com",
    "providers": [
      {
        "id": "default",
        "enabled": false,
        "secret": "base64url-random-secret"
      }
    ]
  }
}
```

规则：

```text
1. 如果没有 scheduler 配置，plugin-core 首次启动时自动生成 default provider。
2. secret 使用 SecureRandom 生成，建议 32 bytes 后 base64url 编码。
3. secret 长期可见，可在 scheduler 调度中心页面复制。
4. 后续可以支持重置 secret，但重置必须写日志或展示明确确认。
5. scheduler 业务数据存 `plugin.runtime.automations.v2` 和 `plugin.runtime.automationRuns.v2`，时间字段统一为 epoch millis。
6. externalHost 只用于生成外部调度器调用地址；为空时使用 loadPublicInfo 的 homeUrl。
7. provider.enabled 只表示外部 tick 入口是否打开，默认 false；它不关闭 plugin-core 内部平台调度。
```

plugin-core 启动后会启动一个轻量内部 tick runner，启动后立即检查一次，之后按分钟边界检查 due automation。
外部 `POST /api/internal/plugin/scheduler/tick` 路由保留为独立调度中心或外部调度器接入点。
内部 runner 不读取外部 tick 开关；外部 tick 由 default provider 的 enabled 状态和 Bearer token 控制。

### 9.7 Cron 支持

第一阶段 cron parser 下放到 `zrlog-plugin-common`，由插件进程和 plugin-core 共享同一套语法与校验规则。

当前实现：

```text
com.zrlog.plugin.common.BasicCronParser
com.zrlog.plugin.common.CronParseException
```

只支持最基础的标准 5 位 Unix cron：

```text
*/5 * * * *
0 3 * * *
```

timezone 默认跟随 plugin-core 进程所在系统时区，仅用于 cron 计算：

```text
ZoneId.systemDefault()
```

实现要求：

```text
1. cron parser 只负责解析表达式和计算下一次执行时间。
2. 对外提供类似 nextRunAt(cron, timezone, now) 的纯逻辑入口。
3. 解析失败时 automations 保存失败，tick 时不执行。
4. parser 实现必须可替换，SchedulerRuntime 不直接依赖具体库 API。
5. 不在 parser 内启动线程或 timer。
6. 第一版只要求能周期执行。
7. 支持 *、*/n 和单个数字值。
8. 暂不支持 L、W、#、复杂 range、复杂 calendar。
9. timezone 不作为第一版配置项，不在页面展示；页面只展示 epoch millis 格式化后的本地时间。
10. plugin-core 不再维护第二份 cron parser，避免插件端预校验和 core 端保存时语义不一致。
```

Quartz 接入决策：

```text
1. 4.0.0 不一步到位接入完整 Quartz Scheduler。
2. Quartz 作为保留方案，不作为 4.0.0 默认实现路径。
3. 4.0.0 按轻量 cron parser 方案执行。
4. plugin-core 仍然使用 website KV + tick endpoint + CapabilityInvoker 执行任务。
5. 后续只有在 JVM 测试、Agent warmup、native image smoke test 都通过后，才考虑把 Quartz CronExpression 作为 cron parser adapter。
6. 如果 Quartz 在 native image 下需要大量反射/资源/序列化配置，继续使用内置轻量 parser 或更小的 cron 解析库。
```

原因：

```text
Quartz 成熟，但完整 scheduler 会引入线程池、JobStore、misfire、序列化和反射边界。
当前阶段只需要 cron nextRunAt，不需要把调度执行权交给 Quartz。
完整 Quartz Scheduler 更适合 persistent job store、cluster、misfire policy、复杂 calendar 和大规模 job 管理。
这些不是 4.0.0 第一阶段目标。
```

后续升级到完整 Quartz Scheduler 的条件：

```text
1. 明确需要秒级 cron、misfire policy、calendar、暂停/恢复 trigger、集群调度等 Quartz 原生语义。
2. 能接受 Quartz cron 6/7 位语法，或完成 5 位 Unix cron 到 Quartz cron 的稳定转换。
3. Quartz Scheduler 在 plugin-core 的 JVM、Agent warmup、native image smoke test 中稳定通过。
4. JobStore 选择与 website KV 的边界明确，不引入第二套不透明的调度状态源。
```

## 10. NotificationRuntime

### 10.1 目标

通知能力标准化，但具体通道由通知插件实现。

### 10.2 边界

plugin-core 负责：

```text
发现通知通道
提供 NOTIFICATION_PUBLISH 标准 socket 入口
记录 notification_delivery
根据 channels 调用 notification channel capability
```

email 插件负责：

```text
SMTP 配置
邮件模板
邮件发送
```

reminder 插件负责：

```text
配置 reminder 使用哪些通知渠道
触发 reminder.due 通知事件
```

### 10.3 NotificationRequest

```java
// com.zrlog.plugin.message.NotificationRequest
public class NotificationRequest {
    private String sourcePluginId;
    private String sourcePluginName;
    private String sourceCapabilityKey;
    private String eventType;
    private String notificationType;
    private List<String> channels;
    private String title;
    private String content;
    private String level;
    private String requestId;
    private String traceId;
    private Map<String, Object> payload;
}
```

字段语义：

```text
sourcePluginId: 发布通知的插件 ID。
sourcePluginName: 发布通知的插件 shortName。
sourceCapabilityKey: 触发通知的 capability，可为空。
eventType: 插件业务事件，例如 reminder.due、backup.failed。
notificationType: 通知分类，例如 reminder、backup、system。
channels: 逻辑通知通道，例如 email、webhook。
level: info、warning、error、success。
payload: 插件业务上下文，不包含通道私有配置。
```

通知发布方式：

```text
插件通过 ActionType.NOTIFICATION_PUBLISH 将 NotificationRequest 发给 plugin-core。
plugin-core 根据 channels 找到 type=notification_channel 且 exposure 包含 notification 的 capability。
plugin-core 通过 CapabilityInvoker 调用具体通知插件。
通知 provider 收到的 capability payload 是标准通知对象，包含 sourcePluginId、sourcePluginName、sourceCapabilityKey、eventType、notificationType、channel、title、content、level、requestId、traceId、payload。
```

### 10.4 通知通道 capability

email 插件声明：

```text
key = notification.email.send
type = notification_channel
exposure = notification
channel = email
```

webhook 插件未来声明：

```text
key = notification.webhook.send
type = notification_channel
exposure = notification
channel = webhook
```

`channels=["email"]` 不直接等价于 capability key。

解析规则：

```text
1. channels 是插件可配置的逻辑通道名称。
2. notification runtime 查找 channel 匹配的 notification_channel capability。
3. 如果多个插件提供同一 channel，调度中心或通知配置需要选择默认 provider。
4. 未知 channel 写 delivery 失败日志，不 silent catch。
```

### 10.5 通知 provider 配置

通知 provider 的选择由 plugin-core 提供配置。

配置沿用现有 `PluginCore.setting`，随 `PluginCore` 一起写入 `plugin_core_db_key`。

建议新增：

```java
public class PluginCoreSetting {
    private SchedulerSetting scheduler;
    private NotificationSetting notification;
}
```

配置示例：

```json
{
  "notification": {
    "defaultProviders": {
      "email": {
        "pluginId": "email-plugin-id",
        "capabilityKey": "notification.email.send",
        "channel": "email"
      }
    }
  }
}
```

默认 provider 规则：

```text
1. 如果存在 email 插件，且声明 channel=email 的 notification_channel capability，plugin-core 默认使用它作为 email provider。
2. 如果已有配置仍然有效，优先保留已有 provider。
3. 如果存在多个 email provider，按插件 shortName/id 稳定排序选择第一个，并在调度中心/通知配置里标记需要确认。
4. 如果不存在 email provider，channels=["email"] 产生 delivery 失败日志，不自动降级。
5. SMTP、模板、发送细节仍由 email 插件自己的配置负责，plugin-core 不读取也不写入 email 插件配置。
```

这个配置属于 runtime 层基础配置，不要求每个业务插件重复保存 provider 信息。

```text
capabilityLabel
providerStatus

reviewRequired
lastDeliveryStatus

```

界面操作：

```text

3. 支持恢复自动选择。
4. 支持发送测试通知，测试结果写 notificationDeliveries。

```

```java
}
```

`PluginRuntimeState.pluginId` 存放插件身份 ID，即 `Plugin.id`。`PluginRuntimeState.pluginName` 存放 `Plugin.name`，只用于展示文案，不参与插件身份判断；为空时可用 shortName 兜底。`instances` 记录同一插件的多个活跃进程或会话实例；`PluginRuntimeInstanceState.instanceId` 存放运行实例 ID，由宿主在运行期生成，不复用 `pluginId`。

运行态页面不再把多个实例合并成一个“插件状态”行，而是直接返回 `PluginRuntimeInstanceView` 列表：一行对应一个进程或 socket 会话实例。默认 `instanceId` 使用可读的规则化名称：当前宿主拉起并能拿到 OS PID 的插件进程使用 `hostname-pid`；没有插件进程 PID 的 socket 会话使用 `ownerId/session-n` 兜底，避免重新引入随机 UUID。`ownerId` 是写入该实例状态的 plugin-core 运行进程 ID，默认同样使用 `hostname-pid`，用于判断本机/远端归属。部署方仍可通过 `zrlog.plugin.runtime.instanceId` 或 `ZRLOG_PLUGIN_RUNTIME_INSTANCE_ID` 显式覆盖当前 plugin-core 的 `ownerId`。

`processId` 是本机宿主可确认的 OS PID，只对当前 plugin-core 以 `Process` 启动并管理的插件进程可靠。FaaS 或跨网插件实例没有本机可控 PID；如果未来插件进程自报 PID，也只能作为远端元数据展示，不能作为本机停止、判活或全局身份依据。页面通过 `local` 区分当前宿主实例和远端/未知实例。

`heartbeatAt` / `leaseExpiresAt` 是运行实例的活性租约。每个持有本地 `IOSession` 的 plugin-core 进程只续写自己 `ownerId` 下的实例；其他进程不能根据本机内存判断远端 session 是否存活，只能读取远端进程写入 DB 的未过期租约。异常退出后实例不再续租，读 runtime state 时会被清理。

`status` 只表示插件生命周期：

```text
starting
initializing
ready
idle
stopping
stopped
failed
```

`activeInvocationCount` 表示当前正在执行的 service/capability 调用数量。`effectiveStatus` 是展示态：当 `activeInvocationCount > 0` 且生命周期已启动时展示为 `executing`，否则等于 `status`。`executing` 不作为持久生命周期状态写入数据库。

`ready` 是内部生命周期状态，含义是插件进程已经连接回 plugin-core 并完成注册；页面展示为“运行中”。`stopped` 在按需加载模式下表示当前没有插件进程，不等同于异常停止；页面展示为“未运行”，用户访问插件页面或运行能力时可重新拉起。

已知限制：当前 `POST /api/runtime-states/stop` 仍是当前宿主、本机进程维度的停止入口。FaaS 多实例或跨网多进程要做到精确停止，需要后续引入广播、租约或远端控制通道；本阶段仅记录实例状态，不把远端实例停止伪装成本机可控操作。

### 11.3 第一阶段要求

第一阶段至少实现：

```text
ensureStarted(pluginId)
markReady(pluginId, session)
markInvocationStart(pluginId)
markInvocationEnd(pluginId)
markFailed(pluginId, error)
idleStop(pluginId)
```

运行态 API 保留插件启动和停止入口，对应：

```http
POST /api/runtime-states/start
POST /api/runtime-states/stop
```

参数：

```text
pluginId=<Plugin.id>
```

`start` 走统一 `ensureStarted(pluginId)`；`stop` 会先检查 active invocation，正在执行任务的本机插件不能被页面直接停止。运行态页面以实例列表为主，停止操作只对 `local=true` 的本机实例开放。

进程进入 `ready` / `idle` 且 `activeInvocationCount = 0` 后，超过预置运行态维护规则中的空闲秒数，会由调度中心回收。

调度中心内置一条系统预置规则：

```text
id = system:plugin-runtime-maintenance
name = 运行态维护
pluginId = __system__
capabilityKey = plugin.runtime.maintenance
cron = */5 * * * *
system = true
deletable = false
targetLabel = 系统任务 / 运行态维护
```

调度任务数据使用 `plugin.runtime.automations.v2`，执行记录使用 `plugin.runtime.automationRuns.v2`。`nextRunAt`、`lastRunAt`、`startedAt`、`finishedAt` 均保存 epoch millis；`timezone` 只表示 cron 计算时区，页面按自己的本地时区格式化时间。

这条规则不是插件能力，不允许删除、停止，不允许修改任务名称或目标；系统任务的展示文案由后端返回，前端不硬编码。调度中心页面只允许编辑执行周期和任务参数。任务参数：

```json
{
  "onDemandEnabled": true,
  "idleStopEnabled": true,
  "idleTimeoutSeconds": 300
}
```

`onDemandEnabled` 是兼容字段，页面表述为互斥的加载策略：按需加载 / 启动时加载。选择启动时加载时，`idleStopEnabled` 会被归一化为 `false`，不再显示或保存有效的空闲回收配置。

运行态维护规则由调度中心统一承接，走与普通定时任务一致的 lease / CAS 抢占流程，FaaS 多实例下也只会有一个 worker 执行本轮清理。规则执行时会清理过期运行实例租约和脏 runtime state；选择启动时加载时，plugin-core 会尝试拉起已启用插件。按需加载不会强制停止已经运行的进程，空闲回收会在满足条件后处理。

plugin-core 内置调度器启动后会立即执行一次到期检查，之后按分钟边界扫描到期任务；`nextRunAt` 到点后不依赖页面刷新或手工点击。

代码结构保留后续扩展：

```text
idleTimeout
maxRunningProcesses
gracefulStop
```

## 21. GraalVM Native Image 兼容要求

### 21.2 Gson model 要求

所有会被 Gson 序列化/反序列化的新增 model 必须满足：

```text

1. GraalvmAgentApplication 必须触发新增 model 的 Gson 序列化和反序列化。
2. PluginNativeImageUtils.usedGsonObject() 必须覆盖 plugin-common 新增 wire model。
3. Agent warmup 必须覆盖 @Capability、@ScheduledCapability 注解读取。
4. Agent warmup 必须覆盖 CAPABILITY_INVOKE、NOTIFICATION_PUBLISH、SCHEDULER_QUERY 和 SCHEDULER_UPDATE 的基础分发路径，其中 SCHEDULER_UPDATE 只验证废弃入口返回失败。
5. 如果新增 classpath resource，必须让 Agent warmup 真实读取该 resource。
6. 如果使用 Quartz CronExpression，Agent warmup 必须覆盖 cron parse、next fire time、非法表达式异常路径。
```

不得新增一个看似正确但实际不会被构建流程使用的静态 native-image 配置文件。

---

### 21.4 注解扫描要求

新增注解：

```text
@Capability
@ScheduledCapability
```

在 native image 下，反射读取注解可能需要保留 annotation metadata。

因此：

```text
1. 注解必须使用 RetentionPolicy.RUNTIME。
2. 被扫描的插件 service 类如果需要反射读取注解，必须确保类和注解元数据在 native image 中可见。
3. 如果当前工程已有 native agent 生成配置机制，新增一次 agent 运行路径，覆盖 capability 注解扫描。
```

新增注解必须类似：

```java
@Retention(RetentionPolicy.RUNTIME)
public @interface Capability {
    String key();
}
```

不得使用：

```java
@Retention(RetentionPolicy.CLASS)
```

---

### 21.5 反射调用限制

现有 `ClientActionHandler` 中存在通过反射调用 action method/service 的逻辑。新增 capability 调用第一阶段应优先复用现有 `@Service` 机制，不要额外引入复杂反射框架。

如果必须反射调用新增 handler：

```text
1. handler 类必须有 public 无参构造。
2. 反射调用的方法必须是 public。
3. 相关类必须进入 Agent warmup 覆盖路径。
4. native 模式必须有测试覆盖。
```

不允许新增依赖 Spring ClassPathScanning、Reflections 这类在 native image 下成本高、配置复杂的扫描机制。

---

### 21.6 资源文件要求

如果新增以下资源文件：

```text
plugin.json
capability.json
default-automations.json
notification-template.html
notification-template.txt
```

必须加入 Agent warmup 的资源读取路径。

第一阶段如果不是必要，不要引入复杂资源文件。优先使用 Java model + website KV 存储，减少 Agent 覆盖压力。

---

### 21.7 Gson 与泛型文档对象

如果使用通用结构：

```java
public class JsonKvDocument<T> {
    private String schema;
    private Integer version;
    private String updatedAt;
    private List<T> items;
    private T data;
}
```

Gson 反序列化时不得直接写：

```java
gson.fromJson(json, JsonKvDocument.class)
```

否则 `items` 会变成 `LinkedTreeMap`，导致后续类型转换问题。

必须使用明确 TypeToken 或专用 document 类。

推荐第一阶段使用专用文档类，减少泛型反序列化风险：

```java
public class PluginAutomationDocument {
    private String schema;
    private Integer version;
    private String updatedAt;
    private List<PluginAutomation> items = new ArrayList<>();

    public PluginAutomationDocument() {
    }
}
```

对应：

```java
PluginAutomationDocument doc = gson.fromJson(json, PluginAutomationDocument.class);
```

不推荐第一阶段大量使用泛型 document。

### 21.8 Native Image 测试要求

Codex 完成实现后，必须至少保证以下命令在 JVM 模式通过：

```bash
./mvnw test
```

如果项目已有 native 构建 profile，必须尝试：

```bash
./mvnw -Pnative -Dagent exec:exec@java-agent
```

如条件允许，再运行项目现有 native package 命令。

如果本地环境无法完整 native build，必须至少完成：

```text
1. 检查新增 Gson model 是否有 public 无参构造。
2. 检查新增注解是否 RUNTIME。
3. 检查新增 model 是否进入 Agent warmup。
4. 检查新增资源是否进入 Agent warmup。
5. 检查没有引入 native 不友好的 classpath scanner。
```

---

### 21.9 禁止事项

在 native image 兼容要求下，Codex 禁止：

```text
1. 禁止新增复杂 classpath scanning 框架。
2. 禁止新增运行时动态代理作为核心调用路径。
3. 禁止让 Gson 反序列化没有无参构造的类。
4. 禁止使用 Agent warmup 覆盖不到的私有构造器/私有方法调用。
5. 禁止新增资源文件但不加入 Agent warmup。
6. 禁止引入只在 JVM 模式可用的反射逻辑。
7. 禁止用泛型 JsonKvDocument 直接反序列化 List<T>。
8. 禁止在 native 模式下依赖 Agent warmup 未覆盖的 annotation metadata。
```

---

### 21.10 最小实现建议

为了减少 native image 复杂度，第一阶段优先采用：

```text
简单 POJO model
public 无参构造
String 日期字段
专用 Document 类
INIT_CONNECT 携带 capability
复用现有 @Service
少量 RUNTIME 注解
不引入复杂扫描库
不新增大量资源文件
扩展 Agent warmup
```

第一阶段不要追求“plugin-core 自动扫描所有类生成 capability”。

插件进程内由 `NioClient` 读取注解，并随 `INIT_CONNECT` 注册 capability；plugin-core 只接收注册结果。

推荐优先级：

```text
INIT_CONNECT 注册 capability > 插件进程内简单注解读取 > 复杂 classpath 扫描
```

目标是：

```text
JVM 模式能跑，Native Image 模式也能稳定跑。
```
